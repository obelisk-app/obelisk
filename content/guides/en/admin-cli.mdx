## Where moderation lives

In the old hybrid Obelisk, admins lived in a Postgres table and an admin web panel called an HTTP API to issue bans. In the new Obelisk, **all of that is on the relay**. Moderation isn't a feature of the app — it's a feature of NIP-29.

That means:

- Any NIP-29 client can show the same admin list, member list, kicks, and bans.
- Switching clients doesn't change permissions.
- Switching relays *does* — moderation is per-relay-per-group, because the relay is the enforcer.

## The events that matter

NIP-29 defines a small set of event kinds. The ones that affect moderation:

| Kind  | What it is                                |
|-------|-------------------------------------------|
| `9007`  | Group create — first signer becomes admin |
| `9000`  | Put user (add member)                     |
| `9001`  | Remove user (kick)                        |
| `9005`  | Delete event (mod-side message removal)   |
| `9002`  | Edit metadata                             |
| `39000` | Group metadata (replaceable)              |
| `39001` | Admin list (replaceable)                  |
| `39002` | Member list (replaceable)                 |

Every action a moderator takes is one of these events, signed by them, timestamped by the relay. There's no hidden moderation log — the events *are* the log.

## The roles

- **Admin** — listed in the latest `39001` event. Can edit metadata, add or remove members, promote or demote other admins, kick, and delete messages.
- **Member** — listed in the latest `39002` event. Can post messages.
- **Everyone else** — read-only at most, depending on the group's metadata flags (`public` vs `private`).

In Obelisk's UI you see these roles surface as badges next to names, "kick" and "ban" buttons in the right-click menu, and an admin section in the group settings. Under the hood, every click publishes one of the events above.

## What an admin actually does

- **Promote a member to admin** — publish a new `39001` adding their pubkey.
- **Kick someone** — publish a `9001` event with their pubkey. The relay then refuses their messages.
- **Delete a bad message** — publish a `9005` event referencing the offending event ID.
- **Rename or rebrand the group** — publish a `9002` (edit metadata).

All of these go through Obelisk's UI today. There is no separate admin panel because there's nothing to "panel over" — the events go straight to the relay.

## What about banning across relays?

NIP-29 bans are per-relay. If someone is banned from your group on relay A, nothing stops them from creating the same group ID on relay B and posting there. This is by design — relays are sovereign. If you care about portable bans, run your own relay (or a relay set you control) and replicate group state across them.

## Where the admin CLI went

Earlier versions of Obelisk shipped a Node CLI (`scripts/admin-cli`) that let agents drive the HTTP admin API with their own nsec. With the API gone, the CLI is gone too. The same outcome — agents managing groups end-to-end — is now achievable directly with `nostr-tools` or any Nostr SDK: load an nsec, build the NIP-29 events, publish to the relay. We may publish a thin helper for this; track the issue tracker.
